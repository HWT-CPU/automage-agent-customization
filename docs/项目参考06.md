# 项目参考 06：Agent Service Toolkit

## 项目链接

https://github.com/JoshuaC215/agent-service-toolkit

## 项目简介

`agent-service-toolkit` 是一个用于运行 AI Agent 服务的开源工具包，重点在于把 Agent 以服务化方式部署，包含 FastAPI 服务、LangGraph Agent、工具调用、状态管理、长期记忆、人类介入等能力。

AutoMage-2 不仅需要 Agent prompt，还需要稳定地和后端 API、数据库、鉴权、任务队列联动。因此这个项目适合作为生产级 Agent 服务化的参考。

## 可以作为我们项目开发参考的内容

### 1. Agent 服务化架构

AutoMage-2 中 Hermes-Agent 不能只停留在 prompt 文件层面，还需要：

- 对外暴露服务入口
- 接收飞书回调
- 调用后端 API
- 维护任务状态
- 处理异常和重试
- 记录日志和执行链路

`agent-service-toolkit` 可以参考它如何把 Agent 包装成可运行、可部署、可观测的服务。

### 2. FastAPI 与 Agent 的结合

AutoMage-2 后端会提供 API，Agent 侧也可能需要自己的服务入口，例如：

- 飞书事件回调入口
- Agent 初始化入口
- 测试触发入口
- Cron 手动触发入口
- 健康检查入口

可以参考该项目中 FastAPI 与 Agent 工作流的结合方式，设计 Hermes-Agent 的服务壳。

### 3. 状态管理和长期记忆

AutoMage-2 的任务不是一次性对话，而是跨天、跨人、跨部门的持续流程。

可以参考该项目的状态管理方式：

- 每个会话或任务有状态
- Agent 执行过程可恢复
- 长期记忆可被工作流调用
- 人类介入点可被保留

这对 Dream 机制、日报链路和任务队列都很重要。

### 4. Human-in-the-loop 人类介入

AutoMage-2 中有明确的人类介入场景：

- Manager 超权限问题上推人工
- Executive Agent 生成 A/B 方案后老板确认
- 鉴权失效通知管理员
- 数据异常需要人工修正

可以参考 `agent-service-toolkit` 的人类介入设计，避免 Agent 全自动误操作。

### 5. 与后端 API 联调的参考

胡文涛需要封装 5 个核心 API：

- `POST /api/v1/agent/init`
- `POST /api/v1/report/staff`
- `POST /api/v1/report/manager`
- `POST /api/v1/decision/commit`
- `GET /api/v1/tasks`

可以参考该项目中的服务化调用、工具调用、状态恢复和错误处理方式，提升 Skill 的工程质量。

## 和 AutoMage-2 的区别

`agent-service-toolkit` 是通用 Agent 服务化模板，不专门面向企业组织管理，也不内置飞书日报、三级组织权限、Schema 契约或 Dream 决策。

AutoMage-2 需要在此基础上设计自己的：

- 组织结构表
- 员工日报表
- 部门汇总表
- 决策日志表
- 任务队列表
- 飞书交互流程

因此，这个项目适合作为 Agent 服务化、FastAPI 接入、状态管理和生产部署的参考。

## 建议优先参考的部分

- Agent + FastAPI 服务结构

- 状态持久化设计
- Human-in-the-loop 节点
