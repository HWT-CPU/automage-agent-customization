# 项目参考 08：OpenClaw

## 项目链接

https://github.com/openclaw/openclaw

## 项目简介

`OpenClaw` 是一个开源个人 AI assistant / Agent 框架，目标是让用户在自己设备或服务环境中运行一个可持续工作的 AI 助手。

它支持通过用户已有的沟通渠道与 Agent 交互，并且具备插件化、渠道接入、多平台运行、长期运行等 Agent 框架特征。

对 AutoMage-2 来说，`OpenClaw` 的重点参考价值不是“业务逻辑”，而是：

```text
如何把一个 Agent 接到企业 IM / 飞书入口，并让员工通过消息平台和 Agent 交互。
```

这和会议纪要中“一线员工每人配备开箱即用框架（Hermes/OpenClaw）”的设计强相关。

- `mem0` 参考记忆层和 Dream 机制，但不是 OpenClaw 或 Hermes。
- `Agent Service Toolkit` 参考 Agent 服务化，但不是 OpenClaw 或 Hermes。

`OpenClaw` 官方项目则直接对应设计要求里的 `OpenClaw` 框架，因此是更强相关的参考项目。

## 可以作为我们项目开发参考的内容

### 1. OpenClaw 作为员工交互入口

AutoMage-2 的岗位级 Agent 不应该要求员工直接使用命令行或开发工具。

更合理的入口是：

```text
员工 → 飞书 / IM → OpenClaw → Hermes Agent → Skill → 后端 API
```

OpenClaw 可以参考为岗位级 Agent 的交互层：

- 接收员工消息。
- 接收飞书卡片回调。
- 把员工输入转成 Agent 可处理的事件。
- 把 Agent 输出转成飞书消息或卡片。
- 作为员工和 Agent 之间的稳定入口。

### 2. Feishu / Lark 渠道接入

AutoMage-2 初期设计里，员工和老板侧都优先通过 IM 呈现。

OpenClaw 有 Feishu / Lark 渠道相关文档和接入思路，可以参考：

- 飞书机器人如何接入。
- 群聊和私聊如何区分。
- 消息事件如何转发给 Agent。
- Agent 回复如何发回飞书。
- 企业 IM 环境下如何处理用户身份。

这对胡文涛要做的飞书交互脚本非常关键。

### 3. 渠道适配层设计

AutoMage-2 不应该把飞书逻辑直接写死在 Hermes prompt 或业务 Skill 中。

更好的方式是参考 OpenClaw，把渠道层单独抽出来：

```text
Feishu Channel / IM Channel
 ↓
事件标准化
 ↓
Agent Runtime
 ↓
Skill / Tool
```

这样后续如果从飞书换成企业微信、钉钉或内部 IM，只需要替换渠道适配层，不需要重写 Agent 模板。

### 4. 员工事件标准化

岗位级 Agent 会遇到多种员工事件：

- 员工提交日报。
- 员工查询任务。
- 员工回复催填消息。
- 员工补充问题。
- 员工确认任务完成。

可以参考 OpenClaw 的事件入口思想，把不同来源的消息统一成内部事件，例如：

```text
DAILY_REPORT_SUBMITTED
TASK_QUERY_REQUESTED
REMINDER_ACKED
TASK_COMPLETED
AUTH_FAILED
```

再由 Hermes / Skill 层处理这些事件。

### 5. OpenClaw 与 Hermes 的分工参考

在 AutoMage-2 中，可以把两者分工定为：

```text
OpenClaw = 员工入口、飞书渠道、事件接收、消息发送
Hermes = Agent 大脑、岗位 prompt、Skill 调用、Schema 修正
```

也就是说：

- OpenClaw 不负责业务决策。
- OpenClaw 不直接做部门汇总。
- OpenClaw 不替代 Hermes 的 prompt 和 Skill。
- OpenClaw 只负责把外部交互稳定地送进 Agent 系统。

这样可以避免 OpenClaw 和 Hermes 职责重叠。

### 6. 安全边界参考

OpenClaw 作为外部入口，会接触用户消息、飞书凭证和工具调用，因此需要重点参考它的安全边界设计。

AutoMage-2 中建议：

- 飞书密钥不写入 `user.md`。
- OpenClaw 只保存渠道凭证和回调配置。
- Hermes 只拿到当前用户上下文和必要工具。
- 后端 API 负责最终权限校验。
- 所有写库动作必须经过后端接口。

对应原则是：

```text
OpenClaw 负责接入，Hermes 负责判断，后端负责校验，数据库负责事实。
```

### 7. 长期运行和自托管参考

岗位级 Agent 需要每天自动运行，而不是临时启动一次。

OpenClaw 的长期运行 / 自托管思路可以参考：

- Agent 服务如何常驻。
- 连接断开后如何恢复。
- 多渠道消息如何保持一致。
- 如何在用户自己的环境中部署。
- 如何让 Agent 在企业内网或本地环境中运行。

这与会议纪要中提到的“本地化部署”“国企贴牌一体机业务”方向一致。

## 对胡文涛任务的具体参考

胡文涛负责 Hermes-Agent 客制化时，可以把 OpenClaw 当作岗位级 Agent 的外部入口层。

### Staff Agent 可参考

- 通过 OpenClaw 接收员工日报消息。
- 通过 OpenClaw 发送 18:00 日报卡片。
- 通过 OpenClaw 发送 20:00 二次催填。
- 员工提交后，把内容交给 Hermes 解析和 Schema 校验。
- Hermes 调用 `post_daily_report` 写入后端。

### Manager Agent 可参考

- 通过 OpenClaw 向部门群推送汇总结果。
- 通过 OpenClaw 接收中层人工补充意见。
- Manager Agent 只读取自己部门的日报数据。
- 超权限事项通过 OpenClaw 通知人工或上级。

### Executive Agent 可参考

- 通过 OpenClaw / 飞书推送 A/B 决策卡片。
- 老板点击按钮后，OpenClaw 接收回调。
- Hermes / Executive Agent 调用 `broadcast_strategy`。
- 后端写入 `decision_logs` 和 `task_queue`。

## 和 AutoMage-2 的区别

`OpenClaw` 是通用 AI assistant / Agent 框架，不是专门为企业三级组织决策设计的系统。

AutoMage-2 还需要自己实现：

- Staff / Manager / Executive 三级 Agent 模板。
- `user.md` 岗位个性化配置。
- 数据库契约存储。
- `schema_v1_staff` 和 `schema_v1_manager`。
- 后端 API 与权限校验。
- Dream 决策机制。
- 任务队列和日报闭环。

因此，OpenClaw 适合作为：

```text
IM 入口层 + 渠道适配层 + 事件转发层 + 长期运行 Agent 外壳
```

不适合直接照搬为 AutoMage-2 的完整业务系统。

## 一句话总结

`OpenClaw` 可以作为 AutoMage-2 岗位级 Agent 的“交互入口和运行外壳”参考：

```text
员工在飞书里操作 → OpenClaw 接收事件 → Hermes 理解岗位任务 → Skill 调后端 API → 数据库沉淀契约数据
```

它能帮助我们更快落地员工侧 IM 交互、飞书入口、事件转发和长期运行能力。
