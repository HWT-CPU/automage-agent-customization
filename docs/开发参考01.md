# 开发参考 01：岗位级 Agent 如何使用 Hermes / OpenClaw

## 设计背景

会议纪要中对岗位级 Agent 的设计要求是：

> 一线员工每人配备开箱即用框架（Hermes/OpenClaw），需填写 `user.md`（岗位职责、输入输出、个性化内容）并与后端联调。

这个设计可以理解为：给每个一线员工配一个可运行的 AI 助手底座，员工只需要填写 `user.md`，系统就能生成该员工的岗位级 Agent。

推荐思路是：

```text
Hermes 做 Agent 内核，OpenClaw 做交互 / 自动化入口。
```

---

## 1. Hermes 与 OpenClaw 的分工

### 1.1 Hermes：岗位级 Agent 的“大脑”

Hermes 主要负责：

- 读取 `user.md`，识别员工身份、岗位职责、输入输出要求。
- 加载岗位级 prompt / `agents.md`，让 Agent 明确自己是岗位级执行节点。
- 调用 Skill，例如提交日报、获取任务、调用后端 API。
- 执行 Self-correction，例如 Schema 校验失败后自动修正再提交。
- 维护上下文，例如当前任务、当天进度、历史输入。

简单说：

```text
Hermes 负责“这个员工 Agent 应该怎么思考、怎么执行任务”。
```

### 1.2 OpenClaw：岗位级 Agent 的“手和入口”

OpenClaw 更适合作为：

- IM 入口：接入飞书 / Lark / 其他消息平台。
- 自动化入口：把外部事件、定时触发、消息回调转给 Agent。
- 工具调用层：让 Agent 可以通过安全工具执行动作。
- 员工交互界面：员工不用接触代码，只在飞书中填日报、收任务、点按钮。

简单说：

```text
OpenClaw 负责“员工怎么和 Agent 交互，以及 Agent 怎么触达员工”。
```

---

## 2. 推荐整体架构

岗位级 Agent 的推荐链路如下：

```text
员工
 ↓
飞书 / IM / OpenClaw
 ↓
岗位级 Agent（Hermes）
 ↓
Skill 层
 ↓
AutoMage-2 后端 API
 ↓
数据库
```

也就是说：

- 员工在飞书中填日报。
- OpenClaw 接收飞书事件。
- Hermes 根据员工的 `user.md` 和岗位模板理解内容。
- Hermes 调用对应 Skill。
- Skill 调用后端接口。
- 后端写入数据库。

---

## 3. `user.md` 的作用

`user.md` 是每个员工 Agent 的个性化配置文件。

它应该包含：

- 身份信息：`user_id`、`node_id`、`role_id`、部门、直属上级。
- 岗位职责：这个员工每天负责什么。
- 输入来源：他从谁那里接任务，接什么类型的信息。
- 输出要求：每天要提交什么日报、什么结构化数据。
- 权限边界：只能提交自己的日报，只能获取自己的任务。
- 个性化内容：岗位术语、业务背景、常见问题、KPI。

核心逻辑是：

```text
同一个 Hermes 模板 + 不同 user.md = 不同员工的岗位级 Agent
```

这样可以做到岗位级 Agent 的开箱即用。

---

## 4. 岗位级 Agent 的每日流程

### 4.1 18:00 日报提醒

OpenClaw 或定时器触发：

```text
发送飞书日报卡片 → 员工填写 → Hermes 解析 → Skill 提交日报
```

调用后端接口：

```text
POST /api/v1/report/staff
```

### 4.2 20:00 二次催填

系统检查谁还没有提交日报。

如果员工未提交：

```text
OpenClaw / 飞书机器人再次提醒
```

Hermes 可以根据 `user.md` 生成更贴近岗位的话术，例如：

```text
今天还有哪些客户跟进没有记录？
是否有需要上级支持的问题？
明天计划是否明确？
```

### 4.3 员工获取任务

员工询问：

```text
我今天还有什么任务？
```

Hermes 调用：

```text
GET /api/v1/tasks
```

然后只返回该员工自己的待办。

### 4.4 日报格式错误时

如果后端返回 422：

```text
Schema 不符合要求
```

Hermes 不应该直接失败，而是要：

- 读取错误信息。
- 自动修正字段。
- 必要时追问员工。
- 再次提交。

这就是岗位级 Agent 的 Self-correction。

---

## 5. Skill 设计建议

岗位级 Agent 至少需要以下 Skill。

### 5.1 基础 Skill

- `agent_init`：启动时调用 `/api/v1/agent/init`。
- `fetch_my_tasks`：获取自己的任务。
- `post_daily_report`：提交员工日报。
- `check_auth_status`：检查鉴权是否失效。

### 5.2 飞书 / OpenClaw 交互 Skill

- `send_daily_report_card`：发送日报卡片。
- `send_reminder_card`：二次催填。
- `handle_card_callback`：处理员工提交的表单。
- `notify_admin_auth_failed`：鉴权失败时通知管理员。

### 5.3 容错 Skill

- `retry_with_backoff`：接口超时后按 1s、2s、4s、8s、16s 重试。
- `schema_self_correct`：根据后端错误修正日报格式。

---

## 6. Hermes 模板设计建议

岗位级 Hermes 模板要明确限制职责边界。

建议模板中包含如下约束：

```text
你是岗位级 Staff Agent。
你的职责是帮助员工完成任务记录、日报提交、任务查询。
你只能访问当前 user_id 的任务和日报。
你不能读取其他员工数据。
你不能做部门级汇总。
你不能代替 Manager 或 Executive 做决策。
```

岗位级 Agent 的目标不是成为万能助手，而是成为：

```text
岗位 SOP 执行助手
```

否则容易出现 Agent 越权、乱汇总、乱决策的问题。

---

## 7. OpenClaw 接入方式

OpenClaw 可以优先承担三个角色。

### 7.1 飞书消息入口

员工不直接操作 Hermes，而是在飞书里：

- 收日报卡片。
- 填日报。
- 查看任务。
- 收提醒。

OpenClaw 负责把这些事件转给 Hermes。

### 7.2 事件分发器

OpenClaw 接收到事件后，判断事件类型：

```text
员工提交日报 → 调 Staff Agent
员工查询任务 → 调 fetch_my_tasks
老板点击决策 → 调 Executive Agent
鉴权失败 → 通知管理员
```

对于岗位级 Agent，只处理员工相关事件。

### 7.3 安全工具网关

不要让 Hermes 直接拿所有系统权限。

更稳妥的方式是：

```text
Hermes 发起意图
OpenClaw / Skill 层执行具体工具
后端做权限校验
```

这样即使 Agent prompt 出错，也不会越权访问数据。

---

## 8. MVP 推荐落地顺序

MVP 阶段不要把 Hermes 和 OpenClaw 都做得太复杂。

### 第一阶段：Hermes 优先

先做：

- 三个 Agent 模板。
- `user.md` 注入。
- Skill 调后端 API。
- 日报提交。
- 任务获取。
- Schema 校验重试。

这样能先跑通核心闭环。

### 第二阶段：接 OpenClaw / 飞书

再做：

- 飞书日报卡片。
- 二次催填。
- 卡片回调。
- 老板决策按钮。
- 异常告警。

这样可以避免一开始被飞书和自动化细节拖住。

---

## 9. 关键注意点

- 数据库才是事实源，Agent memory 不能替代数据库。
- 权限必须由后端校验，不能只靠 prompt 限制权限。
- `user.md` 不放密钥，密钥应放环境变量或配置中心。
- Cron 只能有一个主控，避免 Hermes 和 OpenClaw 同时发提醒导致重复推送。
- 岗位级 Agent 不能做部门决策，只负责自己的任务、日报和执行反馈。
- OpenClaw 不要变成另一个大脑，否则会出现 Hermes 和 OpenClaw 逻辑冲突。

---

## 10. 一句话总结

在这个设计里，最合理的用法是：

```text
Hermes = 每个员工 Agent 的大脑和 Skill 运行层
OpenClaw = 飞书 / IM / 自动化事件入口
user.md = 员工岗位个性化配置
后端 API = 权限和数据事实源
数据库 = 全系统契约数据中心
```

这样既能做到每个员工一个开箱即用 Agent，又不会让系统变成混乱的多 Agent 直接互聊。
