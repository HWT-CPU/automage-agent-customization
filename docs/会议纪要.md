# 会议纪要 

## **会议主题** 

AI公司项目开发规划会——聚焦“岗位SOP转化”与“三级AI架构落地”

## **核心目标** 

将传统不可替代岗位转化为业务SOP，通过**三级AI agent架构**（岗位级-部门领导-公司领导）实现AI中枢驱动的数据决策，提升组织透明度与效率，最终达成“组织架构数字化、去中心化”。

## **一、系统架构与通信设计**

### 1. 三级架构 

- **岗位级agent**：一线员工每人配备开箱即用框架（Hermes/OpenClaw），需填写`user.md`（岗位职责、输入输出、个性化内容）并与后端联调。 
- **部门领导层节点**：读取下属agent的契约数据，整理业绩、处理边界决策（权限内自主决策，超权限推至人工）。 
- **公司领导层agent**：IM推送带决策项内容（如ROI目标拆解方案），老板仅需选方案/补充意见，负责信息收集与传达。 

### 2. 通信路径 

- 摒弃长轮询/WebSocket，采用**数据库存储契约数据**（一线员工每日工作、产出、决策），按需读取，降低通信压力。 
- 数据流向：一线员工→数据库→部门领导汇总→公司领导决策→反向拆解任务至一线，形成闭环。

## **四、关键讨论与决议** 

### 1. 工期与需求 

- **工期**：两周内完成MVP（里程碑三），分期推进，同步媒体沟通。 
- **需求文档**：先出系统架构图，全站与产品协同3天内交付MVP文档（含边界功能），确认后开发。 

### 2. 与飞书的核心区别 

- 飞书：会议级复盘，仅作参考文档；本项目：**数据闭环**（一线→中层→老板全流程透明），推动“传话型中层”转型为“Hcase处理者”，实现组织去中心化。 

### 3. 技术路径与用户真实性 

- **技术路径**：优先复用屏幕记忆开源项目（如LJuly）自动生成Schema；若不可行，改为IM填表单（事件驱动，含数字签名责任划分）。 
- **用户真实性**：依赖数字签名划分责任，通用型MVP不限定业务场景（覆盖所有电脑办公场景）。 

### 4. 其他决议 

- **数据库**：本地部署（考虑国企贴牌一体机业务），技术选型（Postgre/MySQL）待定。 
- **客制化**：开仓库分三级模板，参考首个agent可视化文档，含初始化定义、任务边界、Scale设计。
