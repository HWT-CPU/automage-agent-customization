# 项目参考 01：openhanako

## 项目链接

https://github.com/liliMozi/openhanako

## 项目简介

`openhanako` 是一个个人 AI Agent 框架，目标是把一个 AI 助手接入多个聊天平台，并让它具备记忆、技能调用、定时任务和文件监听等能力。

它更偏向个人 AI 助手 / 多平台聊天 Agent，不是企业组织中枢系统，因此不能直接照搬到 AutoMage-2，但它在 Agent 组织、Skill 机制、Cron 调度和飞书接入方面，和胡文涛负责的 Hermes-Agent 客制化任务高度相关。

## 可以做参考的点

### 1. Agent 运行框架

AutoMage-2 需要三级 Agent：

- Staff Agent：一线员工执行节点
- Manager Agent：中枢汇总节点
- Executive Agent：老板决策节点

`openhanako` 中 Agent、Session、Model、Preferences、Skill 等模块的拆分方式，可以作为 Hermes-Agent 模板仓库的工程结构参考。

### 2. Skill 机制

胡文涛需要实现多个 Skill：

- `post_daily_report`
- `fetch_my_tasks`
- `analyze_team_reports`
- `generate_manager_schema`
- `delegate_task`
- `dream_decision_engine`
- `broadcast_strategy`

`openhanako` 的 Skill 管理方式可以参考：

- Skill 如何注册
- Skill 如何被 Agent 调用
- Skill 如何读取配置
- Skill 如何和外部服务交互

### 3. 飞书 / 多平台机器人接入

胡文涛需要实现飞书交互脚本，包括：

- 日报卡片推送
- 二次催填
- 管理层汇总推送
- 老板侧决策按钮
- 鉴权失效告警

`openhanako` 支持飞书以及其他聊天平台，可以参考其多平台消息桥接思路，尤其是消息收发、机器人事件处理、平台适配层的组织方式。

### 4. Cron 定时任务

AutoMage-2 中有明确的定时任务：

- Staff Agent 每日 18:00 推送日报卡片
- Staff Agent 每日 20:00 二次催填
- Manager Agent 每日 21:00 汇总部门日报
- Manager Agent 每日 22:00 异常预警（待确认）

`openhanako` 支持 Cron 和 heartbeat，可以参考它如何组织定时任务、周期性检查、任务触发与状态维护。

### 5. 记忆与偏好管理

AutoMage-2 的 Dream 机制需要做日报汇总、决策推理和任务拆解。

`openhanako` 的 memory / preference 相关能力，可以作为 Dream 机制或用户个性化上下文管理的参考，但需要根据企业级组织结构重新设计。

## 和 AutoMage-2 的区别

`openhanako` 更偏个人助手，核心是多平台聊天、个人记忆和技能扩展。

AutoMage-2 更偏企业组织 AI 中枢，核心是：

- 员工日报结构化上报
- 部门级汇总
- 老板级决策
- 数据库存储契约数据
- 权限分层
- 任务反向拆解

因此，`openhanako` 适合作为胡文涛任务的工程参考，而不是完整产品原型。
