# 项目参考 04：Feishu / Lark OpenAPI MCP

## 项目链接

https://github.com/larksuite/lark-openapi-mcp

## 项目简介

`lark-openapi-mcp` 是飞书 / Lark 官方 OpenAPI MCP 工具项目，用于把飞书开放平台能力暴露给 AI Agent 或 MCP 客户端调用。

AutoMage-2 中胡文涛需要做飞书交互脚本，包括日报卡片、二次催填、老板决策按钮、消息回调和鉴权失效告警。这个项目可以作为飞书侧能力接入的重点参考。

## 可以作为我们项目开发参考的内容

### 1. 飞书能力工具化

AutoMage-2 需要把飞书能力封装成可被 Agent 调用的工具，例如：

- 发送日报卡片
- 推送二次催填消息
- 发送部门汇总
- 发送老板决策卡片
- 接收按钮点击回调
- 给管理员发送异常告警

`lark-openapi-mcp` 的核心价值是把飞书 OpenAPI 工具化。胡文涛可以参考它如何组织飞书 API 调用、工具声明、参数定义和返回结果。

### 2. 飞书权限与凭证管理

飞书应用接入必须处理：

- App ID
- App Secret
- access token
- 事件订阅 token
- 权限范围
- 用户身份和 open_id

AutoMage-2 后续要在企业环境中部署，飞书权限非常关键。可以参考官方 MCP 项目的凭证配置方式和权限边界提示。

### 3. Agent 调用飞书 API 的方式

AutoMage-2 的 Agent 不应该直接把飞书逻辑写死在 prompt 里，而应该通过 Skill 或工具函数调用。

可以参考这个项目的 MCP 思路：

- 把飞书 API 封装成独立工具
- Agent 只需要知道工具名称和参数
- 工具层负责处理鉴权、请求、错误和返回

这和胡文涛要做的 Python Skill 插件非常一致。

### 4. 飞书消息与卡片交互

AutoMage-2 中飞书是早期主要交互入口。

可以参考飞书开放平台能力，设计这些卡片：

- Staff Agent 日报填报卡片
- Staff Agent 二次催填提醒
- Manager Agent 部门汇总卡片
- Executive Agent A/B 决策按钮卡片
- 系统异常告警卡片

重点参考：

- 卡片字段如何组织
- 按钮点击如何回调
- 如何区分用户身份
- 如何把用户输入转成后端 API 参数

### 5. 与 AutoMage-2 Skill 的对应关系

可以将飞书能力包装成以下 Skill：

- `send_daily_report_card`
- `send_reminder_card`
- `send_manager_summary_card`
- `send_decision_card`
- `handle_card_callback`
- `notify_admin_auth_failed`

这些 Skill 再被 Staff / Manager / Executive 三类 Agent 调用。

## 和 AutoMage-2 的区别

`lark-openapi-mcp` 主要解决飞书 API 工具化问题，不负责三级 Agent 架构、数据库契约、Dream 决策或组织权限模型。

AutoMage-2 需要在此基础上补齐：

- 三级 Agent 模板
- Schema 校验
- 数据库读写
- Cron 调度
- 决策闭环
- 企业组织结构权限

因此，这个项目适合作为飞书交互脚本和飞书工具封装的参考。

## 建议优先参考的部分

- 飞书 OpenAPI 工具封装方式
- 凭证和权限配置方式
- Agent 调用飞书工具的模式
- 消息发送和事件回调设计
- 飞书工具与 Skill 层的解耦方式
