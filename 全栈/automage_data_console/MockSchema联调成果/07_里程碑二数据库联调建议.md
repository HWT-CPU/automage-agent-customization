# 07 里程碑二数据库联调建议

## 1. 当前数据库是否可以支撑里程碑二 Mock 闭环

- **结构层面**：已具备 28 张业务表及 JSONB/payload 承载能力，可支撑「本地 Mock + 可选真实落库」的里程碑二验收。
- **API 层面**：里程碑二 **Mock 文件闭环** 可不依赖真实 API；若验收包含「真实写入验证」，则仍需后端最小链路（见第 6 节）。

## 3. 当前需要后端确认的表

- `staff_reports` 与 `work_records` / `work_record_items` 的主从与幂等键（按 org+user+date？）。
- `manager_reports` 与 `summaries` 的权威读源与版本策略。
- `task_queue` 与 `tasks` 的状态迁移（入队 → 转正 → 关闭）。
- `decision_records` 与 `decision_logs` / `agent_decision_logs` 的分工与保留周期。

## 4. 当前字段缺口

- Mock 映射规则中标记为「需要新增字段 / 接口转换」的条目约 **1** 条（详见 `04_Mock字段-真实数据库映射表.xlsx` 的「字段缺口清单」Sheet）。

## 5. 当前索引缺口

- 建议复核：**(org_id, department_id, report_date|record_date)** 上的 Staff 查询索引；**(assignee_user_id, status)** 上的任务列表索引；**summary_source_links(source_*)** 上的溯源索引。

## 6. 当前接口实现优先级（从高到低）

1. `POST /api/v1/report/staff` + `GET /api/v1/report/staff`
2. `GET /api/v1/tasks` + `PATCH /api/v1/tasks/{task_id}`
3. `POST /api/v1/report/manager` + `GET /api/v1/report/manager`
4. `POST /api/v1/decision/commit` + `POST /api/v1/tasks`
5. `POST /api/v1/agent/init`
6. `POST/GET /api/v1/incidents`
7. `POST /internal/dream/run`

## 7. Staff 日报主表建议

**建议主落点：`staff_reports`（壳 + payload/meta）**；结构化行项同步写入 `work_records` / `work_record_items` 以保持里程碑一兼容。

## 8. Manager 汇总主表建议

**建议主落点：`manager_reports`**；跨层检索与来源链接用 `summaries` + `summary_source_links`。

## 9. Executive 决策主落点建议

**建议主落点：`decision_records`（含 payload）**；过程与审计用 `decision_logs` / `agent_decision_logs`。仅在需要与历史 summary 合并展示时，才额外写 `summaries`。

## 10. 任务草案与正式任务建议

**建议：`task_queue` 承载草案/排队；`tasks` 承载正式任务**；分配与事件用 `task_assignments` / `task_updates`。
